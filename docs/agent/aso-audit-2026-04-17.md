## Sanity Check on the Draft Copy (`aso-appstore-listing.md` §1)

Reviewed against the framework; it's directionally right but has three items to tighten:

1. **"sideline" in the default keyword field is a rejection risk.** Apple's guideline 2.3.7 explicitly prohibits competitor names in keywords. Enforcement is inconsistent — many apps get away with it — but for a first submission where you do NOT want a rejection, remove `sideline` from the default keyword field. Keep it in the per-CPP keyword fields where legally-grey angles are lower-stakes. Replace `sideline` with `forwarding` or `callerid`.
2. **"AI" duplication between new name and new subtitle** (noted above). Keep as-is for brand consistency; acknowledge the 2-char inefficiency.
3. **Description in the draft is very close to the Play Store description.** That's fine — Apple will not penalize that — but cross-platform duplication means one weakness in the copy shows up everywhere. Specifically: the draft (like Play Store) leads with features ("Get a verified business phone number — and let an AI assistant handle calls for you…"). A persona-led hook would likely convert better: *"Stop missing calls on the job. VeraDial gives your business a verified phone number and an AI assistant that handles the calls you don't have time for."* Worth testing later via the description in v1.1 (description changes need a new app version — cannot be metadata-only updated).

Otherwise the draft is defensible and a major upgrade over current live state.

## Concrete Recommendation — Patch Now, or Wait?

**Wait for approval, then push a metadata-only update. Do not PATCH while `WAITING_FOR_REVIEW`.**

Reasoning:
- The release type is `MANUAL`, so approval does not mean public. You control when v1.0 goes live.
- Editing a version that's queued for review can reset the review queue position or flag the submission for closer inspection. Not a guaranteed problem, but unnecessary risk with no upside given MANUAL release.
- **After approval**, Apple allows a metadata-only update that goes through a separate, faster review track (typically 12–24 hours, sometimes same-day).
- The full §1 listing (name, subtitle, keywords, description) all require a metadata update because name/subtitle/keywords are locked to a version.
- **Exception:** promotional text can be updated at any time without review. You could PATCH just promotional text now as a low-risk quick win — it'll populate the product page the moment the app is released.

### Suggested sequence

1. **Now:** PATCH **promotional text only** via ASC API. (Optional, low-risk.)
2. **Wait:** Let Apple review v1.0. Approve with current minimal listing.
3. **On approval, before release:** Push full metadata update via ASC API — name, subtitle, keywords, description. Submit metadata review.
4. **After metadata approved:** Create the 6 CPPs from `aso-appstore-listing.md` §2.
5. **Release v1.0** with the upgraded listing + CPP URLs ready to hand to paid campaigns.
6. **1–2 weeks post-release:** start PPO Test #1 (first screenshot variants, §3).
